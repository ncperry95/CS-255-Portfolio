# CS-255-Portfolio
Portfolio repository for CS 255 – System Analysis and Design. Includes the DriverPass business requirements document and system design documentation demonstrating requirements gathering and UML modeling.

Portfolio Reflection – DriverPass Project
Project Summary

The DriverPass project was completed for a client named DriverPass, a company seeking to improve pass rates for students taking their driving exams. DriverPass wanted a system that would provide online practice exams and scheduling for behind-the-wheel training. The system needed to support multiple users, including students, instructors, administrators, and management. The goal was to design a web-based system that allowed students to register, take practice tests, schedule lessons, and track progress, while administrators could manage users, appointments, and system data.

What I Did Well

One area I performed particularly well in was translating the client’s interview transcript into clear, structured business requirements. I was able to identify both functional requirements (such as user registration, scheduling, reporting, and role-based access) and nonfunctional requirements (such as security, usability, and system availability). I also created UML diagrams that clearly modeled system interactions and workflows. These diagrams supported the written documentation and demonstrated how the system components would work together.

What I Would Revise

If I could revise one part of my work, I would improve the depth of the nonfunctional requirements section. While I identified important aspects like security and performance, I would strengthen it by adding more measurable criteria, such as specific response times, uptime percentages, and data protection standards. Adding more concrete metrics would make the system design more precise and implementation-ready.

Interpreting User Needs

To interpret the user’s needs, I carefully reviewed the DriverPass interview transcript and extracted recurring themes and priorities. I identified pain points, such as low student pass rates and limited access to structured preparation tools. I then mapped those needs into system features, ensuring that each requirement supported the client’s business goals. Considering user needs is critical in system design because the system ultimately exists to solve their problems. If a system does not align with user expectations or workflows, it will not be effective regardless of how technically sound it may be.

My Approach to Designing Software

When designing software, I begin by understanding the problem from the user’s perspective. I focus on gathering requirements, identifying stakeholders, and defining system boundaries. I then use modeling techniques such as UML diagrams, use case descriptions, and activity diagrams to visualize system behavior before implementation. In the future, I would continue using structured requirement gathering, iterative refinement, and stakeholder feedback to improve system clarity and usability. I would also emphasize measurable requirements and early validation to ensure alignment between business goals and technical design.
